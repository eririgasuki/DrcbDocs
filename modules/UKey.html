# 注意点&可能的扩展改造

发放UKEY的时候，需要传入发KEY箱箱号。文档和大部分厂商都是0号发放箱，1号回收箱。但实际也存在1号发放箱，2号回收箱的厂商。

所以获取箱号时一定要根据类型 Type （1-发放;2-回收）来判断。

let result = await UKeyDispenser.getCardUnitInfoAsync()
let unitNo = result.CardUnit.find(e => e.Type === 1).Number

无论最终交易结果是发放还是回收UKEY，在UKEY从KEY箱移动到卡道中时，就需要更新UKEY数量。 调用U008接口，UKEY数量减一。（曾因C端P端都调用数量减一导致数据异常，P端已取消调用）

let result;
try {
  result = await UKeyDispenser.dispenseCardAsync({
    Present: false,
    Number: unitNo
  });
  await RequestHelper.sendAsync(new U008());
} catch (error) {
  console.log(error);
}

读取到的UKEY码实际不是文档中写的String,实际是Number，而且带了一个ASCII的空格\u0000

result = await UKeyReader.readAsync();
this.submitAsync({
  USBKEY_NO: result.toString().trim() //"4838000001817\u0000"
})

这个ASCII的空格在页面中展示时一般不会出现问题，但在生成PDF电子凭证时会无法解析导致异常，所以会提前处理slice(0, -1)。

和上面类似，也要先找到回收箱CardUnit.find(e => e.Type === 1).Number箱号才能进行回收。

东信厂商的UKEY设备模块由于设计问题，在用空盒子（重量较轻）测试时，经常容易卡主或掉落导致故障。尽量少用东信机具测UKEY。

在报错OCCUPIED时，通常是因为设备中有UKEY卡主或掉落在某处。需要手动拿出，或用外设工具调用Reset（Eject）取出。
